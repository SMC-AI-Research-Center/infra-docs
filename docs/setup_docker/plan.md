# 주피터 컨테이너 시스템 업그레이드 및 배포 계획

---

## [1단계] 테스트 및 환경 검증

가장 리소스 부담이 적은 환경에서 신규 이미지를 빌드하고 데이터 이관 시나리오를 검증합니다.

- 대상 서버 선정: 저장 용량이 가장 적은 서버 (빠른 백업 및 복구 테스트용)
- 데이터 안전 확보 (Backup):

    ```bash
    tar -czvf workspace_backup_$(date +%Y%m%d).tar.gz /workspace
    ```

- 데이터 구조 재편 (Path Restructuring):
  - 신규 볼륨 마운트 구조 적용을 위해 기존 데이터를 하위 디렉토리(`home/`)로 이동
  
    ```bash
    mkdir -p /workspace/8888/home /workspace/8889/home
    mv /workspace/8888/- /workspace/8888/home/ 2>/dev/null
    mv /workspace/8889/- /workspace/8889/home/ 2>/dev/null
    ```

- 지식 자산화: 시행착오 과정에서 발생한 트러블슈팅 사례 및 최종 확정 명령어를 위키(Wiki) 또는 문서로 저장

---

## [2단계] 안정성 모니터링

테스트 배포 후 실제 업무 환경에서의 호환성을 검증합니다.

- 검증 기간: 배포 후 1주일 (7 Days)
- 주요 모니터링 항목:
  - GPU 할당 및 연산 가속 정상 여부 (`nvidia-smi`)
  - 신규 콘다 환경(Virtual Env) 생성 및 주피터 커널 인식 여부
  - 컨테이너 재시작 시 데이터 및 가상환경 유지 상태 점검

---

## [3단계] 외부망(Esplash) 점진 적용

검증이 완료된 시나리오를 바탕으로 외부 서비스망에 단계적으로 적용합니다.

- 대상: Esplash 서비스 노드
- 적용 방식: 점진적 적용 (Canary Deployment)
  - 전체 노드 중 일부 노드에 우선 적용 후 트래픽 및 에러 로그 모니터링
  - 특이 사항 발생 시 즉시 이전 버전(`.tar` 백업본)으로 롤백

---

## [4단계] 전사 확산 및 완료

- 일괄 적용 조건: 점진 적용 기간 중 사용자 지연 요청(Delay)이나 치명적 이슈가 없을 시 진행
- 최종 조치: - 구 버전 이미지 및 임시 파일 제거 (`docker system prune`)
  - 사용자용 콘다 환경 활용 가이드 배포

## [5단계] 병원망(Splash) 점진 적용

검증이 완료된 시나리오를 바탕으로 외부 서비스망에 단계적으로 적용합니다.

- 대상: splash 서비스 노드
- 적용 방식: 점진적 적용 (Canary Deployment)
  - 전체 노드 중 일부 노드에 우선 적용 후 트래픽 및 에러 로그 모니터링
