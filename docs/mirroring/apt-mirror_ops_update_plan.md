# APT Mirror 운영 업데이트 기획서 (폐쇄망)

## 1. 목적

- 폐쇄망 클라이언트(100.X, 110.X)가 안정적으로 패키지 업데이트를 수행할 수 있도록 APT 미러(151)와 프록시(149)의 운영 기준을 수립한다.
- 보안 패치 반영 속도와 서비스 안정성을 함께 만족하는 업데이트 주기/절차를 정의한다.

---

## 2. 운영 대상 및 역할

- Origin Mirror (151): 실제 패키지 동기화, 원본 저장, Nginx 서빙
- Reverse Proxy (149): 151 접근 불가 대역(110.X) 요청 중계
- Client (100.X/110.X): hosts/sources.list 기준으로 미러 사용

운영 책임(R&R):
- 인프라 운영자: 동기화 실행, 용량 관리, 장애 조치
- 보안 담당자: 긴급 보안 업데이트 우선순위 요청
- 서비스 담당자: 클라이언트 점검 창구 및 영향 공지

---

## 3. 업데이트 정책 (핵심)

### 3.1 권장 주기

- 정기 동기화: 매일 03:00
- 정기 정리(clean): 매주 일요일 04:30
- 무결성/샘플 검증: 매일 05:00
- 전체 점검(용량, 성능, 에러 패턴): 월 1회

### 3.2 예외(긴급 보안 패치)

- CVE 긴급도 높음(High/Critical): 요청 접수 후 2시간 내 임시 동기화 1회 추가
- 임시 동기화 후 대표 클라이언트 1대에서 `apt update` 검증 완료 시 전파 공지

### 3.3 운영 창(서비스 영향 최소화)

- 기본 작업 시간: 03:00~06:00
- 업무 시간대(09:00~18:00)에는 긴급 패치 외 동기화 지양

---

## 4. 표준 운영 절차 (SOP)

### 4.1 사전 점검 (매일)

```bash
# 디스크 여유 확인 (여유 20% 미만이면 경고)
df -h /data

# nginx 설정/상태 확인
sudo nginx -t
sudo systemctl is-active nginx

# 네트워크/이름해석 확인 (필요 시)
getent hosts archive.ubuntu.com ftp.kaist.ac.kr
```

### 4.2 동기화 실행

```bash
# 백그라운드 실행
sudo nohup apt-mirror > /data/mirror/var/mirror.log 2>&1 &

# 진행 모니터링
tail -f /data/mirror/var/mirror.log
```

### 4.3 정리(clean) 실행 (주 1회)

```bash
sudo sh /data/mirror/var/clean.sh
```

### 4.4 사후 검증

```bash
# 미러 인덱스 확인 (151에서)
curl -I http://archive.ubuntu.com/ubuntu/dists/jammy/InRelease

# 대표 클라이언트(100.X, 110.X 각 1대)에서 검증
sudo apt update
sudo apt-cache policy | head
```

## 5. 자동화 구성안

`/etc/cron.d/apt-mirror` 예시:

```cron
# 매일 03:00 동기화
0 3 * * * apt-mirror /usr/bin/apt-mirror >> /data/mirror/var/cron.log 2>&1

# 매주 일요일 04:30 clean
30 4 * * 0 root /bin/sh /data/mirror/var/clean.sh >> /data/mirror/var/cron.log 2>&1
```

운영 팁:
- `clean.sh`는 최초 동기화 이후 생성되므로, 파일 존재 여부를 확인하고 크론 등록한다.
- 로그 파일은 월 1회 로테이션한다.

---

## 6. 모니터링/알림 기준

### 6.1 알림 조건

- `/data` 사용률 80% 이상: 경고
- `/data` 사용률 90% 이상: 긴급
- 동기화 로그에 `ERROR`, `FAILED`, `404` 급증 시 경고
- 149 프록시에서 5xx 증가 시 경고

### 6.2 필수 확인 지표

- 동기화 소요 시간(전일 대비 증감)
- 신규/삭제 패키지 수
- 대표 클라이언트 `apt update` 성공 여부

---

## 7. 장애 대응 시나리오

### 7.1 404 다발 발생

- 151의 Nginx `root` 경로와 실제 파일 경로 일치 확인
- 클라이언트 `hosts` 매핑(151/149) 재확인
- 필요 시 손상 인덱스 제거 후 재동기화

### 7.2 동기화 중단/실패

- 로그에서 실패 저장소 식별
- 해당 저장소만 재시도 후 전체 동기화 재실행
- 장기 실패 시 해당 저장소를 임시 제외하고 공지

### 7.3 저장소 용량 부족

- clean 즉시 수행
- 오래된 배포판/불필요 저장소 제외 검토
- 용량 증설 전까지 동기화 주기 단축(매일 -> 격일) 임시 적용

---

## 8. 변경 관리 및 공지

- `mirror.list` 변경 시: 사전 검토(영향 범위/용량 증가량/필요성)
- 변경 적용 후: 대표 클라이언트 검증 완료 시 운영 공지
- 월간 리포트: 실패 이력, 용량 추이, 긴급 패치 반영 시간 공유

---

## 9. 권장 운영 일정표

- 매일 03:00: apt-mirror 동기화
- 매일 05:00: 대표 클라이언트 검증
- 매주 일요일 04:30: clean.sh 실행
- 매월 첫째 주: 전체 운영 점검(용량/성능/에러)
